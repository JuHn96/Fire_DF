지능형 CCTV 화재 감지 시스템 – 기업프로젝트 회고 요약본 (상세 서술형)

1. 프로젝트 개요

본 프로젝트는 기존에 설치된 CCTV 인프라를 그대로 활용하면서, AI 기반 화재 감지 모델을 통해 화재 징후(불꽃·연기)를 실시간으로 탐지하고, 이를 웹 대시보드와 카카오톡 알림으로 즉시 전달하는 지능형 관제 시스템을 구축하는 것을 목표로 했습니다.

단순히 “모델만 잘 돌아가는 데모”가 아니라, AI · Backend · DB · Frontend · Docker 인프라까지 하나의 서비스로 통합하는 것을 지향했다는 점에서 팀 단위 기업프로젝트 성격에 맞는 실전형 과제를 수행했다고 볼 수 있습니다.

2. 프로젝트 목적 및 기획 배경

관제실 인력은 동시에 수십 개의 CCTV 화면을 모니터링해야 하며, 초기 연기/불꽃을 놓치기 쉽다는 현실적인 한계가 존재합니다. 실제 화재는 초기 1~2분 대응이 피해 규모를 결정하기 때문에, 사람이 모든 화면을 눈으로 감시하는 방식에는 구조적 한계가 있습니다.

따라서 기존 CCTV를 교체하지 않고도, RTSP 스트림을 AI 서버로 끌어와 자동으로 화재를 감지하고, 그 결과를 관리자에게 실시간으로 전달할 수 있는 체계가 필요했습니다. 이 문제의식에서 출발하여, "현장에 설치해도 바로 쓸 수 있을 정도의 프로토타입"을 목표로 기획이 진행되었습니다.

3. 시스템 구성 요소 요약

1) AI 서버 (화재 감지 모델)

  • RTSP로 CCTV 실시간 영상 수신
  • YOLO 기반 화재 감지 모델 적용
  • 화재 의심 프레임 발생 시 Backend로 이벤트 전송

2) Backend (FastAPI 기반)

  • AI 서버로부터 이벤트 수신
  • 화재 단계/상태 로직 처리 (관심/주의/경고 등)
  • MySQL DB에 이벤트 및 로그 저장
  • WebSocket으로 Frontend에 실시간 이벤트 푸시
  • 카카오 알림 발송 모듈 연동

3) DB (MySQL)

  • CCTV 정보 및 이벤트 로그 저장
  • 카카오 템플릿 / 발송 이력 저장
  • 정규화 기반 스키마 설계

4) Frontend (Vue + Nginx)

  • 대시보드: 다중 CCTV 실시간 상태 표시
  • 서브 화면: 단일 CCTV 확대 보기
  • 로그/통계 화면: 필터, 정렬 기능 포함
  • 관리 화면: CCTV 및 카카오 템플릿 설정

5) 인프라 (Docker Compose)

  • AI / BE / DB / FE 전체를 하나의 compose 파일로 통합 실행
  • fire_net 네트워크 기반 컨테이너 간 서비스명 통신
  • 로컬과 서버 모두 동일한 실행 환경 제공

4. 수행 내용 및 주요 결과

AI 파트

화재 데이터셋 조사 및 라벨링 기준 정리, CCTV 환경을 고려한 RTSP 입력 파이프라인 구성, YOLO 기반 화재 모델 적용 등을 진행했습니다. 시연 가능한 수준의 감지 정확도를 확보하는 것이 목표였고, 실제 데모 영상 처리까지 성공적으로 수행했습니다.

Backend 파트

FastAPI 기반 REST API 및 WebSocket 서버 구현, AI → BE → DB → FE로 이어지는 이벤트 처리 전체 플로우를 설계·구현했습니다. 또한 카카오 알림 API 구조를 이해하고 템플릿 기반 전송 체계를 세웠습니다.

DB 파트

이벤트, CCTV, 카카오 템플릿 등 실제 서비스에 필요한 테이블 구조를 설계했습니다. 정규화와 조회 성능을 염두에 두고 스키마를 구성하며, 전체 시스템의 기반이 되는 데이터 모델을 안정적으로 세팅했습니다.

Frontend 파트

관리자 관점에서 직관적인 UI 구성을 위해 대시보드, 로그/통계, CCTV 관리 화면을 설계했습니다. 특히 WebSocket 실시간 이벤트 반영 로직을 구현하여 “화재 감지 → 바로 UI 반영”이라는 실시간성을 확보했습니다.

인프라 파트

Docker Compose를 활용하여 모든 서비스를 통합 실행 가능하도록 구성했습니다. 환경변수(.env), 빌드 경로 지정, 네트워크 구조 등을 설계하며, 팀원 누구나 동일한 환경에서 실행할 수 있도록 표준화했습니다.

결과적으로, “CCTV → AI → Backend → DB/WebSocket → FE → 관리자 → 카카오 알림” 이라는 완전한 서비스형 흐름을 실제 구현 가능한 형태로 만들었다는 점이 가장 큰 성과입니다.

5. 협업 경험과 회고

역할을 명확히 나누되, 각 기능은 서로 연결되어 있기 때문에 API·WebSocket·DB 스키마 기반의 사전 합의가 중요했습니다. 이를 통해 “설계 문서가 있어야 협업이 쉬워진다”는 점을 체감했습니다.

특히 Backend는 AI와 FE의 중심이 되었기에, 이벤트 구조와 필드를 명확히 정의하는 과정에서 팀 이해도가 크게 올라갔습니다.

Docker Compose로 “내 PC에서만 되는 코드” 문제를 해결하고, 팀원 전체가 같은 환경에서 테스트할 수 있어 협업 효율이 매우 높아졌습니다.

6. 아쉬운 점과 향후 개선 방향

실제 소방 기준에 맞춘 정량적 성능 평가(정탐/오탐률)까지 다루지 못한 점이 아쉬움으로 남습니다.

  • 추가 기능: 인명 감지(쓰러짐, 움직임 없음 등)
  • 로그 분석 고도화: ELK Stack 기반 로그 분석
  • 대규모 실환경 적용을 위한 멀티 카메라 확장
  • GPU 자원 최적화 및 지연시간 감소

이러한 확장 작업을 통해 실제 기업/현장 배포 수준의 서비스로 성장시킬 수 있을 것입니다.
